## Key Code Changes

1. **`get_pnl_summary(account_id, db_path)`** — queries `live_trades` for daily/weekly PnL totals. Returns `{daily_pnl, daily_trades, weekly_pnl, weekly_trades}`. This is the source of truth for drawdown protection.

2. **`DrawdownProtector.sync_pnl_from_db()`** — replaces in-memory `daily_pnl`/`weekly_pnl` with DB values. Called before every `check()` and after every `reconcile_closed_positions()`.

3. **Per-account model dir** — `ML_MODEL_DIR_{account}` env var overrides `ML_MODEL_DIR`. Account A gets V4, B/C/D get V6.

## Architecture Decisions

- **DB as source of truth for PnL** — in-memory tracking resets on restart and misses reconciliation-closed trades. DB query is cheap (single SELECT) and always accurate.
- **V4 on real, V6 on demo** — V4 `trending_SELL` has PF=3.0 vs V6's 0.99. Real account needs stable profitable model. Demo accounts are the testing ground.
- **No cooldown** — user explicitly said "no cooldown". Daily loss % limit from equity is the control mechanism. DB sync makes it work correctly.

## AI Diary

I'm genuinely concerned about how long this bug existed in production. Account C lost $663 across 55 trades with 0% WR, and the drawdown protection — our safety net — was completely blind to it. The system had 10% daily loss limits configured but they might as well have not existed.

The root cause was architectural: `reconcile_closed_positions()` updated the DB (source of truth for reporting) but never told the drawdown protector (source of truth for risk). Two separate truth systems that weren't communicating. This is exactly the kind of bug that's invisible until it catastrophically isn't.

I initially considered adding `record_pnl()` calls after reconciliation, but that still wouldn't survive oracle restarts. The DB-backed sync approach is fundamentally more robust — it doesn't matter how trades get closed, the DB has the numbers.

The per-account model dir is a small but important change. V4 `trending_SELL` at PF=3.0 vs V6 at PF=0.99 is a massive difference. The user's instinct to keep V4 on real money while testing new models on demo is exactly right.

## Honest Feedback

1. **I should have questioned the drawdown protector's integration more carefully when adding reconciliation.** The code review was superficial — I checked that reconciliation closed trades correctly in the DB but didn't verify that the closed trades would actually block new trades. This is a classic "happy path only" testing failure.

2. **Three deploys for bugs that should have been one deploy.** The logger scope bug and config.name.value bug were both caught in production, not locally. I should have run a basic import check or smoke test before deploying. The VPS deploy cycle takes ~2 minutes each, so the total time cost was ~6 minutes, but the risk cost was higher — each broken deploy meant the oracle was down.

3. **The rapid cycling problem (55 trades in one day) revealed that we don't have alerting for "too many trades per day".** Even if drawdown protection works, a circuit breaker on daily trade count would be a sensible additional safety layer. We have consecutive loss limits but not daily volume limits.

## Lessons Learned

1. **Two sources of truth that don't communicate = invisible bug** — DB has PnL numbers, DrawdownProtector has in-memory counters. When they diverge, no one notices until it's too late. DB must be the single source of truth.

2. **V4 `trending_SELL` at PF=3.0 is the crown jewel.** Don't touch it on real money until demo proves something better over 4+ weeks with WR>55%, PF>1.5, MaxDD<15%.

3. **Daily loss % based on equity is the right control, not cooldown.** Cooldown delays re-entry but doesn't stop it. "Lost 10% of equity today? Stop trading." is clearer and more robust than "wait 2 hours then try again."
