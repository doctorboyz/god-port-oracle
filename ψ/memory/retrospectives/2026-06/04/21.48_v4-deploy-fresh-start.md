## Honest Feedback (100+ words, 3 friction points)

**1. Deploy verification is weak.** I deployed to VPS, saw "healthy" status, and assumed it was working. It wasn't until I specifically checked the model loading logs that I found the `_loss` error. The deploy script should include a smoke test that verifies model loading and ML filter health, not just container health.

**2. The column case bug was preventable.** The pipeline had multiple functions expecting different column casing — `load_timeframe()` returns Title Case, `compute_features_from_candles()` expects lowercase, and `BacktestEngine.run()` expects Title Case. This should have been caught by a unit test, not by a runtime KeyError. The fix works but it's a band-aid; a proper solution would be to standardize on one case convention.

**3. VPS model path confusion.** The docker-compose.yml pointed to `trade_outcome_v2` (later changed to `trade_outcome_v2.best`), the predictor code defaults to `data/models/v4`, and the VPS volume has `trade_outcome_v4`. These are three different path conventions that could have been unified. The env var `ML_MODEL_DIR` bridges them, but it's easy to get wrong.

## Lessons Learned

1. **Always pin ML library versions in Docker to match training environment** — sklearn 1.8.0 trained models break on 1.9.0 due to internal module reorganization
2. **Verify model loading as part of deploy, not after** — Add a health check endpoint or deploy verification step that confirms models load successfully
3. **Column casing should be standardized across the pipeline** — Title Case vs lowercase is a recurring bug source; pick one and enforce it
4. **Volatile regime needs more data** — Even at 0.035 threshold, 154 samples is barely enough. Consider more aggressive data augmentation or longer historical backfill
5. **Fresh start on VPS requires closing both DB and MT5 positions** — The oracle-engine checks DB for open positions, but MT5 has its own state. Both must be synced
