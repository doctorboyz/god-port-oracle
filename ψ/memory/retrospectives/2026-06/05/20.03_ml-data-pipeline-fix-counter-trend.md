## Key Code Changes

1. **`predict_loss_proba()`**: `Optional[float]` → `tuple[Optional[float], Optional[str]]` — now returns `(loss_proba, model_name)`
2. **`get_risk_multiplier()`**: `tuple[float, str]` → `tuple[float, str, Optional[float], Optional[str]]` — now returns `(multiplier, reason, loss_proba, model_used)`
3. **Counter-trend logic**: When H4 overrides D1 and signal is counter-trend:
   - `trend_mult = 0.0` (hard block) unless:
   - BUY + oversold (`band_position ≤ 0.15`) → `trend_mult = 0.3` (mean reversion)
   - SELL + overbought (`band_position ≥ 0.85`) → `trend_mult = 0.3` (mean reversion)

## Architecture Decisions

1. **Return values over instance variables**: Instead of `_last_loss_proba` hack (fragile, race-condition prone), changed return types to include all needed data. Callers unpack what they need.
2. **Trend-following as primary, mean-reversion as exception**: Don't trade against trend unless price is at Bollinger extreme — reduces risk while still capturing mean-reversion opportunities.
3. **Collect before retrain**: Wait for 50+ trades with `ml_loss_proba`/`ml_model_used` data before retraining ML model.

## Honest Feedback

1. **I iterated the counter-trend fix 3 times** — first I made `trend_mult=0.5` the default, then the user said they don't want counter-trend at all, so I changed to 0.0, then they said they want mean-reversion at extremes. I should have asked more clearly upfront: "do you want to allow ANY counter-trend, or only at extreme conditions?" instead of assuming.

2. **The DB bug took too long to trace** — I had to read through multiple files to find that `ml_loss_proba` was initialized to `None` and never updated. A simpler code review would have caught this. The `_last_loss_proba` instance variable pattern is fragile and should be replaced by return values everywhere.

3. **I rationalized about the data being "enough"** — when asked if data was sufficient for retraining, I initially said it was fine (9,860 outcomes), but the user rightly pointed out that `ml_loss_proba` was always NULL. I should have been more honest that the ML metadata was missing until the bug fix today.

## Lessons Learned

- **Return values > instance variables**: When a function computes data that callers need, return it. Instance variables for cross-call communication are fragile and create hidden coupling.
- **Ask for intent before implementing**: When the user says "don't trade against trend", ask "what about at extremes?" before hardcoding the strictest interpretation.
- **Check ALL callers before changing signatures**: Found 3 traders + smoke test + health check that all needed updating for the 4-tuple change.
