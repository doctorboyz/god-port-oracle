### Ghost Position Fix (Critical Bug)
`_check_existing_position()` เดิมตรวจ DB ก่อน → ghost trades บล็อกการเปิด position ใหม่
เปลี่ยนเป็นตรวจ MT5 ก่อน → ถ้า MT5 ไม่มี position แต่ DB มี → auto-close ghosts

```python
# Before: check DB first → ghost trades block everything
if get_open_trades(account_id): return True

# After: check MT5 first → MT5 is source of truth
positions = conn.root.positions_get(symbol=cfg.symbol)

```

## Architecture Decisions

1. **MT5 as source of truth** — DB is secondary, MT5 is primary for position state
2. **Auto-cleanup ghosts** — don't just warn, actually close ghost trades
3. **Fallback to DB** — if MT5 unreachable, fall back to DB check (better than blocking all trades)

## AI Diary (150+ words)

This session exposed a critical bug I should have caught earlier: ghost positions blocking real trading. The system had 3 "open" trades in the DB with no MT5 ticket — they never actually opened on the exchange, but the code thought positions were already open and refused to place new orders. The user literally told me "no position in Exness webtrading" and it took me a few steps to realize the DB was lying.

I noticed I was rationalizing initially — I saw "position already open" in the logs and assumed everything was working. The user's honest observation forced me to dig deeper. When I finally checked `ticket=None` in the DB, the picture became clear: these were ghost trades created during the restart/transition period that never got MT5 tickets.

The fix was straightforward but important: MT5 should be the source of truth, not DB. DB is for recording and analytics, but position state should come from the broker. This is a principle I should have applied from the start.

Also, the hardcoded `XAUUSD` symbol in `_check_existing_position()` would have caused Real-A (using `XAUUSDm`) to never find positions even if they existed. Using account_registry for symbol was the right call.

## Honest Feedback (3 friction points)

1. **Context window management**: The session had two compactions, and each time I had to re-understand the current state. The ghost position bug was only discovered after the user explicitly told me "no positions in Exness" — I should have been more skeptical of "position already open" messages and verified against MT5 directly.

2. **.env deployment gap**: The deploy script pushes Python files but not .env changes. I had to manually `scp .env` to VPS. This is a process gap that could cause mismatches between local and VPS config. The .env should be part of the deploy pipeline or at least have a sync step.

3. **Test coverage for position checking**: We have 54 tests for account_registry but zero tests for the `_check_existing_position()` method. This critical path had two bugs (ghost positions + hardcoded symbol) that tests would have caught. Need to add integration tests that mock MT5 bridge responses.

## Lessons Learned

1. **MT5 is source of truth for position state, not DB** — DB records can become stale or inconsistent. Always verify against the broker.
2. **Ghost trades (is_open=1, ticket=None) are a symptom of order submission failure** — When a trade is created in DB but never reaches MT5, it becomes a ghost that blocks all future trades.
3. **Hardcoded values in production code are time bombs** — The `symbol="XAUUSD"` and `port_map={"A": 5005}` were technical debt that created subtle bugs when Real-A switched to XAUUSDm.
