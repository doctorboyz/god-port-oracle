## Key Code Changes

### 1. X11 Lock Cleanup (Phase 0)
```bash
# Before KasmVNC starts:
rm -f /tmp/.X99-lock /tmp/.X11-unix/X99 2>/dev/null
```
Prevents "Server is already active for display 99" fatal error on container restart.

### 2. MT5 Watchdog (Phase 4)
Background process monitors `terminal64.exe` every 10s. On crash: increments counter, cleans X11 lock, restarts terminal. After 10 crashes: stops watchdog and logs VNC login URL. After 60s stability: resets crash counter.

### 3. Dynamic Account Registry
`AccountInfo` dataclass reads all config from env vars. `get_account_config("A")` returns full config. Adding new account = add env vars + docker-compose service. Zero code changes needed.

## Lessons Learned

1. **X11 lock files persist across container restarts** — KasmVNC refuses to start if `/tmp/.X99-lock` exists. Always clean stale locks before starting VNC services.
2. **Container restart ≠ fresh start** — Docker container restart preserves `/tmp/` state. Files that should be ephemeral can survive and cause issues. Always add cleanup logic for known problematic paths.
3. **Watchdog pattern for Wine processes** — MT5 under Wine crashes frequently (error 10053 = WSAECONNABORTED). A background watchdog that restarts it with crash counting and stability windows is essential for production.
4. **Dynamic configs from env vars > hardcoded maps** — The account_registry pattern (read everything from env vars, auto-allocate ports) is far more maintainable than scattering account configs across 8 files.
