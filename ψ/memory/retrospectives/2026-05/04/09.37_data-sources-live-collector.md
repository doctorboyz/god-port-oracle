## AI Diary

I noticed I was rationalizing the 26% model accuracy by saying "expected with mostly-null sentiment data" — and honestly, that IS the real reason, but I should be more concrete about what's needed. The historical data has 16,562 snapshots where fear_greed_value, gold_bias_strength, and news_sentiment are all NULL. Only the 4 live-collected snapshots have real sentiment data. So the model learned to ignore those columns entirely, and without them, it's basically guessing. The fix isn't tweaking hyperparameters — it's collecting live data with sentiment for weeks, then retraining.

The Finnhub calendar bug was genuinely interesting. The API changed from returning unix timestamps to ISO datetime strings, and nobody documented this. The old code did `datetime.utcfromtimestamp(ts)` where `ts` was `"2026-05-04 00:00:00"` — Python tried to interpret the string as an int and crashed. I caught this by actually testing the API response instead of assuming the docs were correct. The country→currency mapping (US→USD) was another gotcha — Finnhub uses ISO country codes, but our filtering was by currency. Without the mapping, filtering `{'USD'}` returned 0 events because every event had `country="US"`.

## Honest Feedback

1. **Data source modules were written without testing against live APIs.** Calendar.py had a timestamp parsing bug, news.py called a method that doesn't exist on the free tier, and yfinance used a ticker that returns 404. All found by testing, not by code review. The lesson: always test external API integrations against the actual service, not just write code that "should work."

2. **Model accuracy is poor and I kept training without addressing the root cause.** The model is 26% on a 3-class problem (33% is random). The root cause is clear: sentiment columns are NULL in historical data. I should have either (a) excluded sentiment features from v2 training, or (b) collected a meaningful amount of live data first. Instead I trained and reported "26%" without a clear path to improvement.

3. **The live collector loads CSV data on every cycle.** For production use with the MT5 bridge, this is fine — the bridge fetches live data. But for testing/demo with CSV fallback, it re-reads the entire 200k-row CSV every 5 minutes. This works but is wasteful. A future optimization would cache the loaded DataFrame between cycles.

## Lessons Learned

- Finnhub API returns string timestamps (`"2026-05-04 00:00:00"`), not unix ints. Always verify API response format empirically.
- Finnhub uses country codes (US, CA, EU), not currency codes (USD, CAD, EUR). Map explicitly.
- `social_sentiment` is a premium Finnhub endpoint — not available on free tier. Use `hasattr` checks.
- `datetime.utcnow()` is deprecated in Python 3.12+. Use `datetime.now(timezone.utc)`.
- XGBoost model quality depends entirely on data quality. NULL sentiment columns = model ignores them. Need weeks of live data before retraining.
