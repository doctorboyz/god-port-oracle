## Key Code Changes

**Before**: Each operation did its own connect/disconnect via `asyncio.run()`:
```python
bridge = MT5Bridge(...)
candles = bridge.fetch_candles_sync(...)  # connects, fetches, disconnects
spread = bridge.get_spread_sync(...)       # reconnects, fetches spread — FAILS
```

**After**: Single persistent connection for the whole cycle:
```python
bridge = PersistentMT5Bridge(...)
bridge.ensure_connected_sync()             # connect once
try:
    candles = bridge.fetch_candles_persistent_sync(...)  # reuse connection
    spread = bridge.get_spread_sync(...)                 # reuse connection
finally:
    bridge.disconnect()                    # clean up
```

## Honest Feedback

1. **Class hierarchy assumption**: I wrote `MT5Bridge` instead of `PersistentMT5Bridge` because I assumed `ensure_connected_sync` was on the base class. It wasn't. I should have checked `client.py` before writing code, not after the deploy failed. This cost ~5 minutes of unnecessary deploy time.

2. **`logger.debug()` for errors is a silent time bomb**: The root cause was hidden for days because `_get_spread()` caught exceptions with `logger.debug()`. INFO is the default log level in production. Any error handler using `logger.debug()` means failures are invisible. This wasn't my bug, but I should have caught it during the original code review.

3. **First-time deploy quality**: The first deploy failed immediately because I didn't test the class resolution. Even a simple mental check — "does `MT5Bridge` have `ensure_connected_sync`?" — would have caught this. The fix was correct in concept but wrong in detail. I need to slow down and verify class hierarchies before deploying.

## Lessons Learned

- `asyncio.run()` from non-main threads is fragile when doing connect-disconnect-reconnect patterns — prefer persistent connections
- Never use `logger.debug()` for exception handlers in production code — minimum `logger.warning()` for anything that indicates a failure
- `PersistentMT5Bridge.ensure_connected_sync()` handles the event loop creation correctly for thread contexts
- Always verify class hierarchy before calling methods — especially when refactoring across base/derived classes
