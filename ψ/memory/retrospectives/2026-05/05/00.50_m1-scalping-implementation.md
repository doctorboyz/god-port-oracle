## AI Diary (150+ words)

This session felt smooth — the plan from the previous session was detailed enough that implementation was mostly mechanical. The only hiccup was the forward reference issue where TradingMode was defined after Signal in models.py, causing a NameError on import. I should have caught that immediately when adding the field, but it only surfaced when I tested imports. The more annoying bug was the DB migration issue on VPS: CREATE INDEX on columns that don't exist yet in an existing table. I put the index in SCHEMA_SQL which runs as a single transaction, so the CREATE TABLE IF NOT EXISTS skipped the table (already exists) but then the CREATE INDEX tried to reference the new columns that haven't been ALTERed yet. Moving the new-column indexes to the migration function (which runs AFTER schema creation) fixed it. I'm also noticing I keep making the same class of mistake: putting new schema additions in SCHEMA_SQL when they need to be in migration for existing databases. Next time, any column additions to existing tables should go ONLY in migration, and SCHEMA_SQL should only be for brand new tables.

## Honest Feedback (3 friction points)

1. **DB migration pattern is fragile** — Mixing CREATE INDEX in SCHEMA_SQL with ALTER TABLE in migration creates ordering issues. Should have a cleaner pattern: SCHEMA_SQL for new tables only, all column additions go through migration.
2. **Forward reference not caught early** — Added TradingMode to Signal before defining TradingMode. Python isn't forward-reference-friendly for class definitions. Should define enums before models that use them.
3. **rsync + build + deploy cycle is slow** — Each VPS deploy takes ~2 min for build + 30s for deploy. No way to speed this up without Docker layer caching tricks, but it's tedious for debugging.

## Lessons Learned

- When adding columns to existing SQLite tables, indexes on those columns must go in migration (after ALTER TABLE), not in the initial SCHEMA_SQL. CREATE INDEX references columns that may not exist yet.
- Python class definitions are evaluated top-to-bottom. If class A references class B, B must be defined before A. Use `from __future__ import annotations` for type hints, but field defaults still need the class at definition time.
- DB migration on VPS with existing data is a different beast than fresh DB on local. Always test with existing data before deploying.
