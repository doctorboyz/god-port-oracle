## Key Code Changes

### TimeSeriesSplit (fixes CV overfitting)
StratifiedKFold(shuffle=True) → TimeSeriesSplit — CV accuracy dropped from 91-97% to 50-67%. This is the REAL accuracy. Previous numbers were look-ahead bias.

### ML Filter Integration Pattern
```python
if self._ml_enabled and self._ml_predictor is not None:
    features = compute_features_from_candles(candles, direction, ...)
    skip, reason = self._ml_predictor.should_skip(features, regime, direction)
    if skip:
        return {"action": "hold", "reason": reason}
```
Off by default (`ML_FILTER_ENABLED=0`). Opt-in via env var.

### Model Persistence
Changed model_dir from `models/` to `data/models/` so models survive Docker rebuilds (oracle-data volume at /app/data).

## Architecture Decisions
1. **ML filter is opt-in**: `ML_FILTER_ENABLED=1` — not forced on users. Rationale: model accuracy is moderate, user should decide when to enable.
2. **Negative filter, not positive**: Reject trades with high P(LOSS) rather than only allowing trades with high P(WIN). More conservative approach.
3. **Model fallback chain**: regime×direction → direction → regime → overall. Uses most specific model available.
4. **MIN_MODEL_ACCURACY threshold**: 0.45 — only use models with >45% test accuracy. Filters out ranging_SELL (20%), trending_BUY (42%), etc.

## AI Diary

I spent most of this session discovering that the ML models aren't as good as I thought. The 91-97% CV accuracy from last session was fake — just look-ahead bias from shuffling time series data. When I fixed it with TimeSeriesSplit, reality hit: 49-55% accuracy. That's barely better than a coin flip.

The uncomfortable truth is that with 32 features and 1,741 trades, I can't reliably predict trade outcomes. The SELL models do better (58-66%) than BUY models (36-42%), which matches the real-world observation that SELL trades have lower win rate (37.6%) — the model learns to predict LOSS correctly. trending_SELL at 66% is the only genuinely useful model.

I caught myself rationalizing: "the model can still work as a filter." And actually, that IS true — rejecting trades where P(LOSS) > 65% is a valid strategy. But I need to be honest that this is a loss-avoidance tool, not a profit-maximization tool. The original goal of WR 58-65% with these features alone is not achievable right now.

What bothers me is that I kept discovering small bugs: CMF returning Series instead of scalar, price_vs_cloud not encoded, FeatureEngineer not saved. Each one was 5 minutes to fix, but there were so many of them. I should have done a full end-to-end test earlier instead of assuming each component worked in isolation.

## Honest Feedback

1. **The ML models are weak, and that's OK.** I spent too much energy trying to squeeze accuracy out of these models instead of accepting the limitation and building around it. A filter that rejects 30% of bad trades while passing 70% of good ones is still useful. Not every ML project needs to be a breakthrough.

2. **I should have tested end-to-end on VPS after each component, not at the end.** The CMF bug, price_vs_cloud encoding, FeatureEngineer serialization — all of these would have been caught immediately if I'd run the full predict flow right after creating the predictor class. Instead I deployed three times just to fix integration bugs.

3. **The Docker model persistence was an obvious design flaw.** Why did I set model_dir to "models/" when I KNEW the only mounted volume was /app/data? That's not a complex architecture decision — it's basic Docker knowledge. I was rushing to get the code working and forgot to check where the data lives.

## Lessons Learned
- TimeSeriesSplit is mandatory for time series ML — StratifiedKFold(shuffle=True) creates fake accuracy through look-ahead bias
- Always save FeatureEngineer alongside models — feature transforms must be identical between training and prediction
- Docker volumes: anything that needs to survive rebuilds must go in a mounted directory
- Model persistence: joblib.dump() works for sklearn models, pickle files are 500-700KB each
