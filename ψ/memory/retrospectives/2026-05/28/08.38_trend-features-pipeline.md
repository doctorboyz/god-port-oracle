```python
def _compute_h4_trend(h4: pd.DataFrame) -> str:
    if h4 is None or len(h4) < 50:
        return "unknown"
    ema10 = close.ewm(span=10, adjust=False).mean()
    ema50 = close.ewm(span=50, adjust=False).mean()
    return "bullish" if ema10.iloc[-1] > ema50.iloc[-1] else "bearish"
```

## AI Diary

วันนี้เริ่ม session ด้วยความสงสัย — ทำไม train v2 มาแล้วไม่เห็น d1_trend ใน features เลย? ทั้งที่ใน DB schema มีคอลัมน์ `d1_trend` ใน `feature_snapshots` ชัดๆ คำตอบคือโค้ด backfill ตั้งใจ strip มันทิ้ง เพราะมองว่าเป็น "metadata" ไม่ใช่ feature — ซึ่งเป็น architectural decision ที่สมเหตุสมผลตอนออกแบบ แต่กลายเป็น blind spot ตอนเราต้องการ regime-aware ML

สิ่งที่ทำให้หงุดหงิดคือเราใช้เวลา session ก่อนหน้าวิเคราะห์ว่า "ML bias จาก training regime" โดยไม่รู้ว่าแท้จริงแล้ว ML ไม่มีทางรู้ regime เลยด้วยซ้ำ เพราะข้อมูล training ไม่มีฟีเจอร์นั้นตั้งแต่แรก! มันเหมือนเราพยายามสอนให้ model แยกสีทั้งที่ให้แค่ภาพขาวดำ

การแก้ไขวันนี้ไม่ยาก — 5 ไฟล์ แก้ตาม pipeline — แต่สิ่งที่ใช้เวลาคือการไล่อ่านโค้ดทั้งหมดเพื่อทำความเข้าใจว่า features_json มันเกิดมายังไง ตั้งแต่ collector → feature_snapshots → backfill → trade_outcomes → load_data → prepare_features จนไปถึงโมเดล Pipeline มันยาวและมีหลายจุดที่ข้อมูลถูก transform ซึ่งการ strip d1_trend ทิ้งที่ backfill คือ single point of failure ที่เรามองไม่เห็นเพราะมันอยู่ใน comment `# Remove metadata keys` ที่ดู innocuous สุดๆ

ตอนนี้ pipeline พร้อมแล้ว แต่ยังไม่มีข้อมูล bearish D1 จริงๆ เพราะ D1 เพิ่ง flip และ ML ยังบล็อก SELL เกือบทั้งหมด — เราอยู่ใน chicken-and-egg: ต้องการข้อมูล bearish เพื่อเทรนโมเดลที่ดีขึ้น แต่โมเดลปัจจุบันบล็อก trades ใน bearish regime ไว้หมด

## Honest Feedback

1. **Data pipeline ยาวเกิน — จุดเดียวพังทั้งระบบ**: การ strip metadata keys ที่ backfill เป็นการตัดสินใจที่สมเหตุสมผล แต่ไม่มีใครจำได้ว่ามันเกิดขึ้น และไม่มี test ที่ verify ว่า features_json มี feature ที่จำเป็นสำหรับเทรน regime-specific models — single point of failure ที่มองไม่เห็นเพราะมันดู innocuous เกินไป

2. **DB มีฟังก์ชัน duplicate**: `backfill_trade_outcomes()` และ `query_trade_outcomes_for_training()` มี 2 copies ใน db.py เหมือนกันเป๊ะ — technical debt ที่ไม่มีใครกล้าแตะ แต่ก็ไม่กระทบอะไรเพราะ Python ใช้อันหลัง ควร clean up ทีหลัง แต่ไม่ใช่ priority

3. **ไม่มี automated test สำหรับ features_json structure**: ถ้ามี test ที่ assert ว่า features_json มีคีย์ที่คาดหวัง (d1_trend, h4_trend, etc.) หลัง backfill เราจะรู้ตั้งแต่แรกว่า trend features หายไป — ตอนนี้ทุกอย่าง trust ด้วยสายตา

## Lessons Learned

- **"Remove metadata keys" ดู innocuous แต่ fatal**: การ strip ข้อมูลที่ "ไม่ใช่ feature" ออกจาก training data เป็น architectural decision ที่ควร document ไว้ชัดเจนว่าอะไรคือ metadata และทำไมถึงไม่ต้องการ — ไม่งั้นจะกลายเป็น blind spot แบบวันนี้
- **Chicken-and-egg กับ regime-aware ML**: ต้องการข้อมูล bearish D1 เพื่อเทรน แต่โมเดลปัจจุบันบล็อก SELL หมด — ต้องมีกลยุทธ์เก็บข้อมูลช่วง transition เช่น ลด threshold ชั่วคราว หรือเปิด learning mode ใน bearish regime
