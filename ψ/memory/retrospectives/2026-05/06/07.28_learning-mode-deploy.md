## Key Code Changes

**M5ScalpTrader._fetch_candles()**: Was using PersistentMT5Bridge (persistent connection that never worked). Now creates fresh MT5Bridge per call (same pattern as LiveCollector that works reliably).

**Learning mode architecture**: Each trader reads `os.environ.get("LEARNING_MODE", "0") == "1"` in __init__. Signal generators receive `learning_mode` param from traders. When enabled:
- All confidence thresholds bypassed (recorded, not blocking)
- Session gates bypassed (logged as "learning: asian session trade")
- Spread filters bypassed (logged as "learning: no spread data")
- Circuit breakers, cooldowns, news filters bypassed
- Direction threshold lowered to 0.05 (captures very weak signals)

## Architecture Decisions

1. **MT5Bridge over PersistentMT5Bridge**: Persistent connections were failing silently. Per-call connections (connect→fetch→disconnect) are more reliable and match the proven LiveCollector pattern.

2. **learning_mode as env var, not config**: Environment variable allows toggling without code changes. Each component reads it independently — no tight coupling.

3. **Record blockers, don't silently skip**: When learning mode bypasses a filter, it appends `(learning: X)` to the reason string. This preserves the data for post-analysis while letting trades through.

## Honest Feedback (100+ words, 3 friction points)

1. **VPS .env management is fragile and error-prone** — I appended vars with `echo >> .env` three times and got concatenated lines twice. The deploy script's `source .env` wasn't propagating vars to docker compose. Should have a proper env file template or a deploy script that validates all required vars before restarting. This wasted 30+ minutes of rebuild cycles.

2. **I kept assuming the fix worked without verifying** — After each deploy, I checked logs but didn't verify the container env vars until things still weren't working. The pattern was: deploy → check logs → find it's still wrong → check env vars → find the real issue. Should verify env vars FIRST after every deploy.

3. **The M5 Scalp bridge failure should have been caught in testing** — PersistentMT5Bridge was a code-level choice made in the plan that never worked in production. The LiveCollector was already using MT5Bridge successfully — I should have checked what the working code was doing before writing new code that used a different approach. Copy-paste the working pattern first, optimize later.

## Lessons Learned

1. **Always copy the working pattern**: When adding a new component that needs to connect to MT5, use the same connection pattern as the working component (LiveCollector → MT5Bridge). Don't invent a new approach.

2. **Verify env vars after every deploy**: `docker exec <container> printenv VAR` should be the first check after any config change, before looking at logs.

3. **Learning mode is the right approach for phase 1**: Instead of tuning thresholds blind, let everything trade and analyze which factors correlate with wins. The "(learning: X)" tags in reason strings are the key innovation — they let us do post-hoc analysis without losing signal data.

4. **.env file management needs a proper solution**: Consider using a `.env.template` file and a validation script that checks all required vars are set before deploying.
